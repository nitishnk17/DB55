          Introduction                                                            
                                                                                  
          Most database servers have limited RAM to process data. When datasets grow or resources
          are limited, data systems rely on disk storage (of course for other reasons too). However,
          disk I/O is far slower than RAM. This makes smart utilization and minimal disk access critical
                                                                                  
          for implementing efficient and effective data management systems.       
                                                                                  
          In this assignment you will design a query execution engine that operates under a strict
          memory budget, optimizing it for minimal disk I/O and efficient in-memory computations.
                                                                                  
                                                                                  
                                                                                  
          System   Architecture                                                   
                                                                                  
          ┌─────────────────────────────────────────────────────┐                 
          │          Monitor        │                                             
                                                                                  
          │ (spawns processes, sends queries, validates output)│                  
          └──────────────┬──────────────────────┬───────────────┘                 
                 │ FD 5 (query in) │ FD 6 (results / commands out)                
                 ▼         ▲                                                      
          ┌──────────────────────────────────────────────────────┐                
          │         Database (YOU)     │                                          
          │ reads query → accesses disk → outputs result rows │                   
          └──────────────┬──────────────────────┬────────────────┘                
                                                                                  
                 │ FD 4 (commands out) │ FD 3 (block data in)                     
                 ▼         ▲                                                      
          ┌──────────────────────────────────────────────────────┐                
          │        Disk Simulator     │                                           
          │ (block storage, tracks I/O for scoring) │                             
          └──────────────────────────────────────────────────────┘                
                                                                                  
                                                                                  
          To give you a hands-on experience with building data systems, we’ve designed an in-house
          system in Rust comprising three components:                             
                                                                                  
            1. Disk Simulator — simulates a block-addressable storage. It stores raw table files
                                                                                  
               and provides a read/write anonymous space for your database to use as scratch. It
               tracks I/O (reads and writes) access patterns and estimates simulated disk time,
               which contributes to your score (more on this later).              
            2. Monitor — This is the orchestrator. It spawns the disk and database processes,
               sends the query, and validates your output against expected results.

            3. Database — this is what you will implement. It receives a query from the monitor,
               reads table data from the disk, executes the query, and streams the result rows back
               to the monitor.                                                    
                                                                                  
          You typically need to touch only the database/ directory.               
                                                                                  
          File Descriptors                                                        
                                                                                  
                                                                                  
          The monitor maps the following file descriptors for your database process at startup:
                                                                                  
           FD Direction          Purpose                                          
                                                                                  
           3  Read     Receive data from disk simulator                           
                                                                                  
           4  Write    Send commands to disk simulator                            
                                                                                  
           5  Read     Receive data from monitor (query JSON)                     
                                                                                  
           6  Write    Send data to monitor                                       
                       (results/commands)                                         
                                                                                  
          These are already wired up in database/src/io_setup.rs — you do not need to open
                                                                                  
          any files or sockets yourself.                                          
                                                                                  
          Constraints                                                             
                                                                                  
          The monitor enforces the following hard limits on your database process:
                                                                                  
            1. Memory: At least 64 MB virtual address space (RLIMIT_AS), including the stack
               (RLIMIT_STACK). Note, you will not be able to load entire tables into memory as
                                                                                  
               they typically will be of several gigabytes.                       
            2. Single-threaded: No spawning subprocesses (RLIMIT_NPROC = 1).      
            3. No file I/O: Writing files on disk is forbidden (RLIMIT_FSIZE = 0). Use the disk
               simulator’s anonymous space for any scratch storage.               
            4. Time limit: A generous time limit prevents infinite loops. Details will follow soon as
                                                                                  
               assignment progresses.

          AST Node Types                                                          
                                                                                  
          A query is a tree of QueryOp nodes. The five node types are:            
                                                                                  
                                                                                  
            Node                        Description                               
                                                                                  
           Scan   Leaf operator that reads all rows of a table from disk.         
                                                                                  
           Filter A unary operator that passes only rows satisfying all predicates.
                                                                                  
           Projec A unary operator that outputs a subset of columns, optionally renaming them.
                  Preserves row order from its child.                             
           t                                                                      
           Sort   A Unary operator that outputs rows sorted by a list of column ordering
                  specification.                                                  
                                                                                  
           Cross  A binary operator that computes the Cartesian product of two child relations.
                                                                                  
                                                                                  
          Operation Semantics                                                     
                                                                                  
            ●  Project preserves the row order of its child. If a Project appears as an ancestor of
               Sort, the final output will reflect that sort order even though sort happened before
                                                                                  
               projection — this is intentional.                                  
            ●  Filter , Cross and Scan do not guarantee any particular output ordering. You are
               free to produce rows in any order for these operators.             
            ●  An SQL query can be represented as a combination of these five operators arranged
               in the correct tree structure.                                     
                                                                                  
          Example                                                                 
                                                                                  
                                                                                  
          Consider:                                                               
                                                                                  
          SELECT a1 AS id, b2 FROM A JOIN B ON A.a1 = B.b1 WHERE b3 > 0 ORDER BY a2
          ASC, b2 DESC;                                                           
                                                                                  
                                                                                  
          The AST is (innermost executed first):                                  
                                                                                  
          Project(a1→id, b2→b2)                                                   
           └── Sort(a2 ASC, b2 DESC)                                              
             └── Filter(b3 > 0)                                                   
                └── Filter(a1 = b1)                                               
                  └── Cross                                                       
                                                                                  
                     ├── Scan(A)                                                  
                     └── Scan(B)                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  

--- Page 3 ---

                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
          Notice that Sort appears before Project in the tree — this is because a2 would be
          dropped by Project before Sort could use it. Since Project is order-preserving, the final
          output is still sorted correctly.

          Filter Predicates                                                       
                                                                                  
          A Filter node holds a list of predicates; a row passes only if all predicates are satisfied.
                                                                                  
          Predicate = { column_name, operator, value }

          operator ∈ { EQ, NE, GT, GTE, LT, LTE }                                 
                                                                                  
          value ∈ { Column(name) | I32(i32) | I64(i64) | F32(f32) | F64(f64) | String(string) }
                                                                                  
                                                                                  
          When value is Column(name), compare the row’s value in column_name against its
          value in the named column. Otherwise compare against the literal.       
                                                                                  
                                                                                  
          Query Assumptions                                                       
                                                                                  
            1. The query is always valid (well-typed predicates, no missing columns, no ambiguous
               names).                                                            
            2. The two children of a Cross node will never share a column name.   
            3. Strings contain only ASCII characters (a–z, A–Z, 0–9, and some special characters).
                                                                                  
               String comparison is lexicographic by ASCII code point.            
            4. Column names contain no whitespaces.

          Disk Simulator Specification                                            
                                                                                  
          Block Model                                                             
                                                                                  
          The disk is a flat array of fixed-size blocks. The block size is at least 512 bytes (typically
          4096) and is the same for all blocks. Block IDs start at 0. A block is the smallest addressable
          unit of disk.                                                           
                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  

--- Page 6 ---

                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
          The disk is divided into two regions:                                   
                                                                                  
            ●  Read-Only (RO) Region — blocks 0 to anon-start-block - 1. Contains files.
               Write attempts here will fail. Reads to blocks not belonging to any file will fail.
            ●  Read-Write (RW) Anonymous Region — blocks anon-start-block to 2^64 -
               1. Use this as scratch space (intermediate results, sorting buffers, etc.). A block must
                                                                                  
               be written before it can be read.                                  
                                                                                  
              Important: Please read the FAQ on anonymous region usage.           
                                                                                  
          A file on disk is a continuous sequences of blocks. A file starts at its start-block block ID
          and spans its num-blocks number of blocks.                              
                                                                                  
              Important: Files on disk need not be placed sequentially one after another,
              query for start-block and num-blocks of a file to know where exactly it is
                                                                                  
              and how many blocks it spans.                                       
                                                                                  
          Commands                                                                
                                                                                  
          All commands and responses end with \\n unless noted. Always flush after writing
          commands.                                                               
                                                                                  
          Get commands (database → disk):                                         
                                                                                  
          get block-size                                                          
                                                                                  
                                                                                  
                                                                                  
          Returns the block size in bytes as a decimal integer.                   
                                                                                  
          get block <start_block_id> <num_blocks>                                 
                                                                                  
                                                                                  
          Returns exactly num_blocks * block_size raw bytes, which correspond to  
          num_blocks blocks starting from start_block_id. The response does not end with
          \\n.                                                                    
                                                                                  
                                                                                  
          get file start-block <file_id>                                          
                                                                                  
                                                                                  
          Returns the starting block ID of the file as a decimal integer.         
                                                                                  
          get file num-blocks <file_id>                                           
                                                                                  
                                                                                  
          Returns the number of blocks the file spans as a decimal integer.       
                                                                                  
          get anon-start-block                                                    
                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  

--- Page 7 ---

                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
          Returns the block ID at which the anonymous (writable) region starts.   
                                                                                  
          Put commands (database → disk):                                         
                                                                                  
          put block <start_block_id> <num_blocks>                                 
                                                                                  
                                                                                  
          (ends with \\n) Followed by exactly num_blocks * block_size raw bytes (not ending
          with \\n). Writes the num_blocks blocks of data starting at start_block_id. All the
                                                                                  
          blocks must be in the anonymous region (>= anon-start-block). No response is sent.
                                                                                  
          <aside> 💡                                                               
                                                                                  
          Example to put 2 blocks of data at block ID 42 and 43, you would send   
                                                                                  
          put block 42 2\\n<2*block_size number of bytes to put starting from     
          block 42> to disk                                                       
                                                                                  
          </aside>                                                                
                                                                                  
          <aside> 💡                                                               
                                                                                  
          Note: Remember to flush your output.                                    
                                                                                  
          </aside>                                                                
                                                                                  
          Exit (optional):                                                        
                                                                                  
          exit                                                                    
                                                                                  
                                                                                  
                                                                                  
          Terminates the disk simulator cleanly.                                  
                                                                                  
                                                                                  
                                                                                  
          Raw File Format                                                         
                                                                                  
          Each .bin file represents one table. The file size is always an exact multiple of
          block_size.                                                             
                                                                                  
                                                                                  
          Block Layout                                                            
          byte 0                                                                  
                                                                                  
          ┌──────────────────────────────────────────────────────┐                
          │ <Row 1 data><Row 2 data>............................│                 
          │ ....................................................│                 
          │ ....................................................│                 
          │ ....................................................│                 
          │ ....................................................│                 
          │ ....................................................│                 
                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  

--- Page 8 ---

                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
                                                                                  
          │ ....................................................│                 
          │ ....................................................│                 
          │ ....................................................│                 
          │ .................<Row N-1 data><Row N data>.........│                 
          │ ....................................................│                 
          │ ....................................................│                 
          │ .....(unused padding bytes).........................│                 
          │ ....................................................│                 
                                                                                  
          │ ....................................................│                 
          │ ............<row_count[2 bytes, u16, little-endian]>│                 
          └──────────────────────────────────────────────────────┘                
                               byte block_size-1                                  
                                                                                  
                                                                                  
            ●  The last 2 bytes of every block hold row_count as a unsigned 16-bit little-endian
               integer.                                                           
                                                                                  
            ●  Rows are packed starting at byte 0 with no padding between them.   
            ●  Usable data space per block = block_size - 2 bytes.                
            ●  A row is always fully contained within a single block — it never spans two blocks.
                                                                                  
          Row Encoding                                                            
                                                                                  
          Columns are stored sequentially in the order defined by column_specs. Each value is
                                                                                  
          encoded as:                                                             
                                                                                  
            Type              Encoding                                            
                                                                                  
           Int32  4 bytes, signed little-endian                                   
                                                                                  
           Int64  8 bytes, signed little-endian                                   
                                                                                  
           Float3 4 bytes, IEEE-754 LE                                            
                                                                                  
           2                                                                      
                                                                                  
           Float6 8 bytes, IEEE-754 LE                                            
           4                                                                      
                                                                                  
           String UTF-8 bytes followed by a 0x00 null terminator                  
                                                                                  
                                                                                  
          Example                                                                 
                                                                                  
          Schema: (id: Int32, name: String) Rows: (0xDEADBEEF, "hi") and          
          (0x1BADB002, "DBMS")                                                    
                                                                                  
          Encoded bytes (hex):                                                    
                                                                                  
                                                                                  
          EF BE AD DE 68 69 00 02 B0 AD 1B 44 42 4D 53 00

          Monitor Interface                                                       
                                                                                  
                                                                                  
          Receiving the Query                                                     
                                                                                  
          On startup, read one line from Monitor. It is a JSON-encoded Query. Parse it with
          serde_json::from_str. (starter code does it for you).                   
                                                                                  
          Querying the Memory Limit                                               
                                                                                  
                                                                                  
          At any point before producing output, you may ask the monitor for the memory limit:
                                                                                  
            1. Write get_memory_limit\\n to Monitor.                              
            2. Read one line from Monitor. It is the memory limit in MB as a decimal u64.
                                                                                  
          Sending Results                                                         
                                                                                  
                                                                                  
          Once query execution is complete, stream results to Monitor as follows: 
                                                                                  
            1. Send validate\\n — signals the start of output.                    
                                                                                  
                                                                                  
          For each row in the query result (in the correct order), send one line: 
                                                                                  
          <col1>|<col2>|...|<colN>|                                               
                                                                                  
            2. Values are formatted as plain text (integers and floats as decimals, strings as-is).
               Each row ends with a trailing | and a \\n.                         
                                                                                  
                                                                                  
            3. After all rows, send !\\n — signals end of output.                 
                                                                                  
            4. Remember to flush the output.                                      
                                                                                  
            5. At this point your database program can exit cleanly. (i.e. you need to execute only a
               single query).                                                     
                                                                                  
                                                                                  
                                                                                  
          Example — a row (26, "Alice", "9487882379") is output as:               
                                                                                  
          26|Alice|9487882379|                                                    
                                                                                  
                                                                                  
              Note: The monitor guarantees that no result rows contains a | character in any
              string column, so this delimiter is unambiguous.

          FAQ                                                                     
                                                                                  
                                                                                  
            1. Can we modify the starter code? Can we add Cargo packages?         
                                                                                  
               You are free to modify any part of the starter code including the monitor, disk, and
               generator for local testing purposes or addition of additional traits. The final
               submission must produce a database executable that communicates via the
                                                                                  
               protocols described in this document. The starter code already includes minimal
               packages sufficient for the assignment. If you believe an additional crate serves the
               learning objectives of the assignment, post on Piazza.             
                                                                                  
               Takeaway: The database executable comes from your code (monitor + disk +
               database + configs + fdwrapper … all source codes combined). But the monitor and
               disk executable from our own source code, so all we require is that your database
               executable follows the database config provided, and runs using the exact protocols
               specified in this document.                                        
                                                                                  
                                                                                  
            2. Can we use multiple threads?                                       
                                                                                  
               No. The monitor sets RLIMIT_NPROC = 1, which prevents spawning additional
               processes or threads.                                              
                                                                                  
            3. Can we write temporary files to disk?                              
                                                                                  
                                                                                  
               No. The monitor sets RLIMIT_FSIZE = 0. All scratch storage must go through the
               disk simulator’s anonymous block region (blocks at or above anon-start-block).

            5. The monitor says validation failed — how do I debug?               
                                                                                  
               Run your query against SQLite and compare the output line by line. Make sure you
               are applying the trailing pipe trick and that your row order matches (if the query has
               no ORDER BY, any order is accepted — but make sure your expected output file also
                                                                                  
               has the same order). Use disabled: true to isolate a single query config.
                                                                                  
            6. What is the size of anonymous space? Disk crashed by OOM? How is 2^64
               blocks of disk space even possible?                                
                                                                                  
               The disk simulator currently stores the Read-Write Anonymous (scratch) blocks
                                                                                  
               entirely in your system's RAM. To prevent your database from crashing the simulator
               by running out of memory, you must understand how the simulator allocates space:
                                                                                  
                 ○  Virtual Address Space is Massive: You are completely free to use any block
                    ID between anon-start-block and 2^64 - 1.                     
                 ○  Physical Allocation is Lazy (Sparse): The simulator does not allocate
                    memory for the empty space between block IDs. It only allocates
                                                                                  
                    block_size bytes of actual RAM when you write to a new, unique block ID.
            7. Example: Assume anon-start-block is 100, and your block_size is 4KB.
                                                                                  
                 ○  Safe (Sparse Allocation): Your database writes data to block 100 and then
                                                                                  
                    writes data to block 10,000,000. Because you only touched 2 unique block
                    id’s, the simulator allocates exactly 8KB of RAM. It ignores the millions of
                    empty blocks in between.                                      
                 ○  Unsafe (Exhausting Memory): Your database writes data to every single
                    block sequentially from 100 to 1,000,100. You have now touched 1,000,000
                    unique blocks. The simulator will allocate ~4 Gigabytes of RAM.
                                                                                  
            8. The Takeaway: There is no restriction on how high your block IDs go, but there is a
               strict limit on the total number of unique block IDs your program touches. Design your
               database to reuse and recycle scratch blocks (e.g., via a Buffer Pool) to keep your
               memory footprint low. We will announce the hard limit on the total allowed
               anonymous size soon. (it will be at least 10GB per say or even higher depending on
               the size of datasets we decide to use.)
